## § 0 项目背景(上下文无关 AI 必读)

### 0.1 背景假设(已脱敏)

- 项目发起者具备 ML / LLM Agent / 软件工程经验。
- 重度使用 Codex / Claude Code / Cursor 等 AI coding 工具。
- 熟悉 Python / TypeScript 等常见工程栈。
- 关注中文开发者常见场景,例如企业协作工具、国内数据源、国内外 LLM SDK 等。
- 本文档保留产品与架构决策,不保留个人身份、公司、所在地或账号信息。

### 0.2 项目产生的动机

作者在真实使用 Codex 等 AI coding agent 时反复遇到两类具体痛点:

1. **流程中断问题**:任务还没做完就中断,每一步都要用户回复"继续"、"确认"。大部分询问是不必要的噪音。
2. **复杂项目执行力弱**:给一份较完整的产品文档,单个 agent 没法一次做好,总遗漏环节。

作者判断这是整个 coding agent 赛道的共性问题,且目前没有好的解决方案 — Devin / Cursor / Codex 都存在。因此有机会通过一层"编排"来显著改善体验。

### 0.3 作者的执行约束与偏好

这些对决策有直接影响,新接手者必须了解:

**时间预算**:按个人/小团队副业项目约束设计。**v0 的设计目标是一天内做出来**。

**已知执行障碍**(作者自述):想太多不动手、完美主义、过度追求细节、眼高手低、刚开始就想放弃。因此项目架构必须防止陷入过度规划,所有决策偏好"先跑通再优化"。

**核心原则**:一切为了赚钱和正反馈,即使学习/练手阶段也要尽快变现,避免为学习而学习、为完美而拖延。参考《小而美》,做顺风的事,发现没价值及时止损。

**语言偏好**:中文交流,文档和 prompt 可以中英混合,但面向 LLM 的指令尽量用模型训练数据覆盖好的表述方式。

### 0.4 讨论过程简述

本项目经历了多轮讨论,从"想做一个通用 AI 软件自动生成平台"逐步收敛到"消除 babysitting 的轻量编排层"。过程中讨论了:

- 是否应该做 / 商业前景评估(结论:定位调整后可做)
- Agent 数量与角色划分(结论:4 个,按工序不是按拟人角色)
- Skill 系统的时机与来源(结论:v0 不做,后期从失败中提炼)
- 接口冻结的必要性(结论:系统承重柱,硬约束)
- Snippet 复用池(结论:v0.3 引入,是差异化关键)
- Observer / 反思机制(结论:v0.4,不可过早)

## § 1 产品定位:为什么是"消除 babysitting"而不是"自动生成软件"

### 1.1 讨论起点

作者最初设想:"给一份产品文档,自动生成完整可用的软件产品,不需要每次跟用户询问实现逻辑。"

### 1.2 为什么这个最初设想被调整

**市场层面**:"文档到完整软件"是 2024 下半年到 2025 年整个 AI coding agent 赛道的核心战场。Devin、Cognition、Factory、Devlo、MetaGPT、ChatDev、OpenHands、SWE-agent、Replit Agent、v0、Bolt、Lovable、Cursor Composer 等至少 30 个团队在做同一件事,其中多个已融资过亿美金。作者每天只有 3-4 小时,正面竞争不现实。

**技术层面**:
- **需求不完备性**:真实产品文档永远不完整,人类 PM 也写不出完整的。"不询问用户"意味着 AI 脑补,脑补的内容大概率与用户期望不符。行业普遍做法反而是**强化 clarification 能力**。
- **并行协调成本**:多 Agent 并行在 demo 看着酷,实际 merge 冲突、接口错配、重复实现、上下文漂移会吃掉大部分收益。
- **测试-修复循环发散**:没有人在 loop 里,AI 修 bug 经常越修越糟,因为它会"理解错测试意图"然后改代码迎合。

### 1.3 为什么定位调整后依然值得做

作者的关键洞察:**现有产品确实没有一个真正好用的**。Devin 口碑两极、Cursor Agent 中长任务会断、Codex CLI 同样。痛点真实且未被解决。

调整后的定位:**不追求取代人类工程师,不追求完美产出,只追求"比用户直接用单个 Codex 显著更好用"**。具体是:

- 用户可以启动任务后离开 4 小时,回来看到阶段性成果
- 省去反复回复"确认"、"继续"、"不对"的 babysitting
- 即使产出有瑕疵,对用户的时间 ROI 依然是正的

这个定位的优势:
- **评估标准具体可测**:干预次数 / 单次执行时长 / 产出可用率
- **失败模式对用户友好**:跑 2 小时卡住 = 损失 2 小时机器时间,不是用户时间
- **成功率 50% 对用户就是净赚**:因为用户用单 Codex 本来也没有 100%

### 1.4 为什么不选更窄的垂直切入(Chrome 扩展生成器等)

讨论中曾建议作者选择"Chrome 扩展专用生成器"、"Shopify app 专用"等垂直方向。作者的反驳有力:

> 现有产品连通用的都做不好,说明痛点在编排层,不在"领域知识"。我只要做出一个比单 Codex 好用的通用编排,对很多场景都有用,没必要强行垂直。

这个反驳成立。垂直切入适合"技术能力差不多、靠领域知识做差异"的场景;如果核心痛点就是"通用能力不足",垂直化反而是自缚手脚。

### 1.5 最终定位的完整表述

> **一个编排层,让用户使用 AI coding agent(Codex / Claude Code 等)时,能把"启动任务"和"验收成果"之间的所有 babysitting 步骤自动化,从而实现真正的长程无人值守开发。**

## § 2 七条核心思想的完整推理

### 2.1 (a) "比 Codex 更好用,不追求完美"

作者明确拒绝"完美主义"导致的拖延陷阱。评估产品成败不能用"代码是否商用级"这种模糊且高标准的指标,而是用"相对单 Codex 的改善程度"这种相对、具体的指标。

排除的替代表述:
- "生成生产级代码" — 标准太高,作者自己承认一天做不到
- "达到人类工程师水平" — 无法衡量

对架构的约束:产品不需要对产出做过强校验,只要流程能自动推进、结果大致可用。不必引入复杂的代码审查 agent、安全扫描 agent 等(这些是 v0.6+ 的事)。

### 2.2 (b) "长程任务执行能力"

作者观察:Codex 跑 10 分钟就要问一次。这意味着用户不能真正离开,相当于 AI 只是个"打字更快的键盘",没产生时间杠杆。

长任务断掉的真实原因:
1. 上下文窗口污染
2. 目标漂移
3. 错误累积
4. 缺乏外部记忆
5. 产品默认保守

我们的对症方案:
- 每个循环产出结构化摘要,关键节点新开 session
- `spec.md` / `interfaces.md` 作为冻结的北极星
- tester 作为独立角色,早期发现问题
- 文件协议 + snippet 池 + api-probes 作为外部记忆
- manager agent 自动消化默认询问

其中最关键的前置动作是:开发前由 Codex 主导和用户完成结构化澄清,把产品目的、需求边界、技术栈、外部 API、账号/密钥依赖、验收标准、非目标全部固化为文件。这样 developer/tester 不需要在长时间执行后依赖模糊记忆,也不会在开发中反复问同一类问题。

多 agent 接力不是长任务唯一解,但配合文件协议和冻结接口,信息传递损耗可控,且为后续并行保留空间。

### 2.3 (c) "并行处理"

并行是**有条件的**,不是无脑并行。

并行前置条件:
- 模块接口已冻结
- 模块之间无紧密依赖
- 单个子任务时长足够大(>5-10 分钟 agent 时间),否则编排 overhead 吃掉收益

推荐并行粒度:按 vertical slice 并行。v0 不做并行,串行闭环先跑通。

### 2.4 (d) "避免打扰用户"

可以中断用户的三种情况:
1. 业务层面的产品决策
2. 必须用户本人操作(API key、账号密码、支付信息等)
3. 系统长时间卡死无法推进(manager 判断已无法自决)

不能中断用户的情况:
- "是否继续下一步" → 永远继续
- "是否这样实现" → 自己决定并记录
- "要不要加这个功能" → 看 spec,有就做没就不做
- "测试失败了要不要修" → 修,直到改好或卡死

manager 自决原则:
- 有 spec 引用 spec
- 无 spec 但有合理默认 → 用默认 + 记录
- 无 spec 也无默认 → 加入 blockers,继续做别的

### 2.5 (e) "前期调研要做足"

AI 开发错误成本远高于 AI 调研错误成本。调研产出必须是结构化工件:
- 功能清单
- 验收标准
- 技术栈决策(带 why)
- 非目标清单
- 已跑通的 API 样例

API 验证特别重要:先写最小 curl/脚本跑通真实 API,把最新现实 ground truth 注入 context,降低 API 幻觉。

这不是一次性让 AI 自己脑补方案,而是一个开发前 discovery 阶段。Codex 应主动识别高影响不确定性,向用户集中澄清产品目的、使用场景、必须/可选功能、技术偏好、外部依赖、数据来源、验收方式。澄清结果进入 `discovery.md` / `spec.md` / `interfaces.md` / `api-probes/`,再进入开发循环。

### 2.6 (f) "模块化与接口冻结" — 系统承重柱

整个架构的核心承诺(长程执行、可跳过卡壳、可并行、可测试)全部依赖模块解耦。如果接口会变,所有承诺都会失效。

接口冻结的具体实现:
1. researcher 阶段必须产出 `interfaces.md`
2. 文件冻结,开发阶段不允许修改
3. developer prompt 明确禁止修改公开接口
4. 测试基于 `interfaces.md` 契约,不基于实现
5. 确需修改时,打回 researcher 重新评审

### 2.7 (g) "Snippet 复用池"

通用 coding agent 不敢假设用户有什么私有库,只能每次从零生成。我们可以假设,因为服务的是单个用户/团队的重复使用,越用池子越厚。

Snippet 不是传统包,而是给 agent 读的组件说明,包含代码、调用样例、已知坑点、配置模板、真实响应样例。

初始来源:
- 飞书通知(Webhook)
- OpenAI / Claude / 国内大模型 SDK 调用
- AKShare(A 股数据)
- Binance K 线数据
- Remotion 视频导出
- Solidity / Rust 合约调用

## § 3 架构决策

### 3.1 为什么是 4 个 agent 不是更多也不是更少

最终选择 researcher + manager + developer + tester。

researcher 必须独立,因为开发前澄清和调研产出是后续所有 agent 的北极星,职责与 manager 完全不同。researcher 要负责把用户的模糊想法压缩为可执行、可验证、可冻结的开发契约;manager 则负责在契约已存在的情况下推进执行。更多 agent 则容易陷入拟人化角色切换,成本大于收益。

### 3.2 为什么 Agent 按工序不按角色

人格切换不产生新信息,只产生不同语气。工程价值来自产出物的结构化和可验证性。按工序分强制每个 agent 有明确输入/输出契约。

### 3.3 为什么用文件协议不用消息传递

文件协议优势:
1. 可观察
2. 可 checkpoint
3. 可 replay
4. 去对话污染
5. git 友好

代价是增加 I/O 和文件格式设计,但可接受。

### 3.4 为什么主循环是 while 不是事件驱动

同步 while 循环一天能做出来,调试简单,足够支撑 v0。v0.6+ 做并行时可升级为简单 task queue,但不建议过早上 Temporal 等重型工具。

## § 4 关键原则的完整推理

### 4.1 "先跑通再优化"

对抗完美主义和拖延。v0 必须一天出,每期目标聚焦于能跑通,不是跑得好。

### 4.2 "文件协议,不是对话协议"

见 § 3.3。

### 4.3 "接口冻结"

见 § 2.6。这是系统承重柱级别的硬约束。

### 4.4 "打扰用户是贵的"

见 § 2.4。这是产品灵魂。

### 4.5 "Skill 来自失败,不是来自书本"

AI 生成抽象原则类 skill 信息增益极低。真正有效的 skill 是具体、可操作、有反例的,只能从真实失败中提炼。

Skill 价值密度:
- 书本/AI 生成 → 极低
- 自己项目的具体失败案例 → 高
- 跑完几十个真实任务提炼的错误模式 → 最高

v0 阶段不花时间写 skill。系统裸跑,记录失败。v0.4 引入 observer 后,才开始积累高价值 skill。

### 4.6 "复用 > 重写"

见 § 2.7。

### 4.7 "日志是未来的资产"

v0 就要强制存 trace,否则 v0.4 observer 无米下锅。

存储内容:
- 每个 agent 的完整输入和输出
- 每次文件读写记录
- 时间戳
- 最终结果(成功 / 失败 / 用户介入)

## § 5 讨论过但未采纳的想法

### 5.1 垂直切入(Chrome 扩展 / Shopify app 专用)

未采纳原因:作者判断核心痛点在编排层不在领域知识。通用编排做好对多场景有用。

### 5.2 完整的工作流引擎(Temporal 等)

未采纳原因:v0 一天预算容不下。简单 while 循环足以支撑早期功能。

### 5.3 v0 就引入 Skill 系统

未采纳原因:AI 生成的 skill 价值密度极低。v0 时间紧,不值得花在装饰品上。

### 5.4 拟人化 Agent 角色(PM / 架构师 / QA 等)

未采纳原因:角色切换不产生新信息,容易陷入"演戏"。按工序分更有效。

### 5.5 一次性写完整 Skill 库

未采纳原因:书本原则已在 LLM 训练数据中,重复 = 零增益。真 skill 从失败中来。

### 5.6 多 Agent 共识投票 / Chain-of-Verification

未采纳原因:v0 阶段过于复杂,成本不对应收益。tester agent + 用户 diff 审核已经提供基本保障。

## § 6 给新接手 AI 的操作指引

### 6.1 首次启动前必读

1. `PLAN.md` 全文
2. 本文档 § 0 背景 和 § 4 关键原则
3. `TODO.md` 当前真实实现状态
4. 当前 run 目录中的 `progress.md`(如存在)
5. 当前 run 目录中的 `discovery.md` / `spec.md` / `interfaces.md`(如存在)

### 6.2 判断当前应该做什么

```
1. 有 spec.md 和 interfaces.md 吗?
   ├─ 没有 → 进入 discovery/researcher 模式,先澄清再产出契约
   └─ 有 → 继续 ↓

2. 有 progress.md 吗?
   ├─ 没有 → 创建,从 interfaces.md 提取待做模块列表
   └─ 有 → 继续 ↓

3. progress 里有待做模块吗?
   ├─ 有 → manager 模式,分派下一个
   └─ 无 → 进入 tester 模式,跑最终集成测试

4. 有 blockers 吗?
   ├─ 有 → 检查能否自解,不能则汇总给用户
   └─ 无 → 任务可能完成,等用户确认
```

### 6.3 遇到不确定的事怎么办

1. 查 `spec.md` / `interfaces.md` 是否已定义
2. 查 `discovery.md` 是否记录了用户意图和关键决策
3. 查 `lessons.md`(如存在)是否有相关经验
4. 查 `snippets/INDEX.md` 是否有现成方案
5. 按 § 4.4 原则:能自决就自决 + 记录理由
6. 确实需要用户 → 写入 `blockers.md`,继续做别的

### 6.4 修改本文档的规则

- `PLAN.md` 任何修改都应同步更新本文档相关章节
- 新增决策请加入 § 5(如果排除了某方案)或对应章节
- 不要删除已有决策记录,即使决策被后续推翻 — 保留历史推理链是本文档的核心价值
